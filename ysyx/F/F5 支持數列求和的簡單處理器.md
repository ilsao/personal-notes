
蕪湖，是時候來實現一個簡單的處理器啦！

我們來約定一些 sISA 的細節：
- PC 的位寬為 4 位。
- GRP 有 4 個，每個位寬 8 位。
- 支持以下三個指令：

```
 7  6 5  4 3   2 1   0
+----+----+-----+-----+
| 00 | rd | rs1 | rs2 | R[rd]=R[rs1]+R[rs2]         add指令, 寄存器相加
+----+----+-----+-----+
| 10 | rd |    imm    | R[rd]=imm                   li指令, 装入立即数, 高位补0
+----+----+-----+-----+
| 11 |   addr   | rs2 | if (R[0]!=R[rs2]) PC=addr   bner0指令, 若不等于R[0]则跳转
+----+----------+-----+
```

我們將這個小 CPU 稱為 sCPU。

現在，我們先實現一個最簡單，只支持 `li` 的 sCPU。

# 只有一條指令的 sCPU

## 取指

首先需要一個 PC 寄存器，因為我們之前已經實現過寄存器了，對我們來說應該是小菜一碟。

![[ysyx/F/pic/F5/1.png]]

但是，我們需要根據 PC 在存儲器尋找指令，所以我們還需要實現存儲器。

存儲器和寄存器都可以存儲信息，但是存儲器還支持尋址 (addressing)。

我們可以將存儲器看成一個由比特構成的矩陣，矩陣的每個 row 稱為一個存儲字 (word)，地址就是 row 的編號。

row 的數量稱為存儲器的深度，而存儲字的位寬代表存儲器的寬度。我們通常用 `深度 x 寬度` 表示存儲器的規格，例如 `2 x 3` 的存儲器排布如下：

| 地址  | 存儲字                           |
| --- | ----------------------------- |
| 0   | $b_{(0,2)}b_{(0,1)}b_{(0,0)}$ |
| 1   | $b_{(1,2)}b_{(1,1)}b_{(1,0)}$ |

而存儲器又分為只讀存儲器 (Read-Only Memory, ROM) 與隨機訪問存儲器 (Random-Access Memory, RAM)，前者不支持寫入，後者支持。

對於 sISA 來說，沒有指令會訪問存儲器，所以我們實現 ROM 就行。

一個 `2 x 3` ROM 結構如下所示：

![[ysyx/F/pic/F5/2.png]]

上圖左上方的譯碼器又稱為**地址譯碼器**，與地址譯碼器相連的導線稱為**字線(word line)**，每條字線對應一個存儲字。與或門相連的導線稱為**位線(bit line)**，每條位線對應存儲字的一位。

實際上，圖中的地址譯碼器、與門、或門共同構成一個 3 位的 2 選 1 多路選擇器。因此 ROM 的讀操作也可視為從多個存儲字中選擇一個，而 `addr` 就是選擇端。

特別地，ROM 中的存儲信息是直接由高低電平編碼的，所以 ROM 從功能上看也可視為數據端為常數的多路選擇器。

因為我們需要存儲最少 8 條指令 (求和程序所需的指令數)，而每條指令占 8 位，所以我們要實現一個 `8 x 8` 大小的 ROM。但是既然我們 PC 有 4 位，那我們乾脆實現一個 `16 x 8` 的 ROM 好了。

好久之後，我搭好了：

## 譯碼

譯碼分為操作碼譯碼與操作數譯碼，由於現在只需要實現 `li`，我們可以視為任意取到的指令就是 `li`。

對於操作數譯碼，我們只需要解析出 `li` 中的 `rd` 與 `imm` 即可。從電路上來看，這只是些位抽取操作，並不難。

## 執行

`li` 需要將 `imm` 放入 `rd` 中，所以我們需要考慮實現 GPR。

GPR 通常包含多個寄存器，因此 GPR 也支持尋址。

因為 GPR 的本質也是個存儲器，且支持寫入，所以 GPR 也是 RAM。

# 實現完整的 sCPU

首先考慮實現 `add` 指令。在譯碼階段我們必須讀取 `opcode` 判斷當前指令類型，這可以通過譯碼器生成的獨熱碼判斷。

而且，我們還需要考慮操作數。`add` 指令需要讀取 `rs1` 與 `rs2`，寫入 `rd`。也就是說，我們需要為 GPR 添加兩個讀口。

添加後的 GPR 應該包含以下功能：
- 第一個讀口：`raddr1` `rdata1` 
- 第二個讀口：`raddr2` `rdata2` 
- 寫端口：`waddr` `wdata` `wen` `clk` 

最後，執行階段我們要判斷寫入的數據屬於 `li` 還是 `add`，我們可以使用多路選擇器實現此功能。

我們先對 GPR 進行改進，長這樣：

![[ysyx/F/pic/F5/7.png]]

接下來我們考慮重構譯碼階段電路：

![[ysyx/F/pic/F5/8.png]]

因為最終只需要支持三個指令，所以在譯碼階段我刪去了中間寄存器，直接使用 MUX 按照選擇寫入數據。

最後，我們來實現 `bner0` 指令。

注意到 `bner0` 隱含著讀取寄存器 `R[0]`。並且讀取 `rs2` 字段的位與 `add` 指令相同，所以可以復用 `add` 中讀取 `rs2` 的邏輯。

讀出操作數後，`bner0` 需要比較二者是否相等，我們可以通過比較器實現此功能。

注意到，只有當指令為 `bner0` 且操作數不相等時，才將 `PC` 更新為 `addr`，否則 `PC = PC + 1`。

又因為 `bner0` 不需要修改 RAM 的內容，我們需要將 `WEN` 設置無效。

# 重新審視 CPU

我們將數據在 CPU 中流動的路徑與路徑上的元件稱為 CPU **數據通路(data path)**，在前文討論中數據通路中的部件有 GPR、加法器、存儲器等。

當多條指令的數據通路發生衝突，我們需要引進更多的電路控制數據流動以取得正確的輸出。這些額外的電路屬於處理器的控制邏輯，決定控制邏輯的信號稱為控制信號。設計數據路徑與控制信號正是 CPU 設計中最關鍵的兩個步驟。

為了更好的了解 CPU 運行的是否正確，考慮添加一個新的指令 `out rs`。

```
 7  6 5  4 3  2 1  0
+----+----+----+----+
| 01 | 00 | 00 | rs | R[rs]         out 指令，輸出指定寄存器
+----+----+----+----+
```

並且，我們需要準備好我們的測試程序：

```
10001001    # 0: li r0, 9
10010001    # 1: li r1, 1
10100001    # 2: li r2, 1
10110010    # 3: li r3, 2
00010111    # 4: add r1, r1, r3
00101001    # 5: add r2, r2, r1
11010001    # 6: bner0 r1, 4
01000010    # 7: out r2
11100010    # 8: bner0 r2, 8
```

由於我嘗試過了 1 加到 10，所以上述程序是 `1 + 3 + ... + 9`。
