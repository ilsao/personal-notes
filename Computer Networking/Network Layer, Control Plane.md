
# Routing Algorithms

路由選擇算法分為：
- **集中式路由選擇算法(centralized routing algorithm)**：知道完整的網絡開銷與節點間的連通性後，通過計算這些結果而取得最低開銷路徑。這種算法通常被稱為**鏈路狀態(Link State, LS)** 算法。
- **分散式路由選擇算法(decentralized routing algorithm)**：路由器以迭代、分布式的方法計算出最低開銷路徑，沒有節點知道網絡的完整信息。這種算法通過相鄰路由器間的交換報文而交換資料，可能更適合路由器直接交互的控制平面。

路由選擇算法還分為：**靜態路由選擇算法(static routing algorithm)** 與**動態路由選擇算法(dynamic routing algorithm)**。

靜態路由選擇算法中的路由變化較為緩慢，通常是人工調整。

相反，動態路由選擇算法可以週期性的對網絡拓樸變化進行響應，但是更容易受到路由選擇循環或路由震盪之類問題的影響。

第三種路由選擇算法的分類方法還有：負載敏感、負載遲鈍。

**負載敏感算法(load-sensitive algorithm)** 中，鏈路開銷會跟隨擁塞水平上升而上升。相反，**負載遲鈍算法(load-insensitive algorithm)** 中則不會。

如今的路由選擇算法通常是負載遲鈍的。

## The Link-State (LS) Routing Algorithm

在 LS 中，每個路由器是如何知道網絡拓樸的狀態？答案是使用**鏈路狀態廣播(link state broadcast)** 算法。

ok 呀，本書略過很多細節後直接開講 Dijkstra。

我們定義以下符號：
- $D(v)$：到目前迭代為止，從源節點 $u$ 到目的節點 $v$ 的最低開銷路徑的開銷。
- $p(v)$：以源節點到 $v$ 的最低開銷路徑來看，到達 $v$ 的前一個節點。(經由此節點，下一跳就是 $v$)
- $N'$：節點組成的子集。若從源節點到 $v$ 的最低開銷路徑已經確定，則 $v$ 會被加入 $N'$ 中。

Dijkstra 分為 Initialization 與 Loop 兩個階段：

```
/* Initialization */
N' = {u}
for all nodes v
	if v is a neighbor of u
		D(v) = c(u, v)
	else
		D(v) = INT_MAX
		
/* Loop */
Loop
find w not in N' such that D(w) is a minimum
add w to N'
update D(v) for each neighbor v of w and not in N'
	D(v) = min(D(v), D(w) + c(w, v))
until N' = N
```

## The Distance-Vector (DV) Routing Algorithm

**距離向量(Distance-Vector, DV)** 算法是一種迭代(持續與鄰居交換信息)、異步(不要求節點間同步計算)、分布式(每個節點都需要與鄰居交換信息)的算法。

我們得先知道著名的 Bellman-Ford 方程：

$$
d_{x}(y)=\text{min}_{v}\{c(x,v)+d_{v}(y)\}
$$

其中，$v$ 取自 $x$ 的鄰居節點集合。

以下，我們令 $N$ 為 $x$ 的鄰居節點集合。

在 DV 算法中，每個節點 $x$ 都會維護以下信息：
- 對每個鄰居 $v$ 的開銷 $c(x,v)$。
- 節點 $x$ 的距離向量，保存了到網絡中其他節點的距離列表。$\mathbf{D}_{x}=[D_{x}(y):y\in N]$ 
- $x$ 的每個鄰居 $v$ 的距離向量。$\mathbf{D}_{v}=[D_{v}(y):y\in N]$ 

在 DV 算法中，每個節點不定時的給他的鄰居發送自己距離向量副本。當 $x$ 收到他的鄰居 $w$ 發的新距離向量，他就保存 $w$ 的距離向量，然後如下使用 Bellman-Ford 方程更新自己的距離向量：

$$
D_{x}(y)=\text{min}_{v}\{c(x,v)+D_{v}(y)\}\quad \forall v\in N
$$

若 $x$ 有更新自己的距離向量，則 $x$ 應向他的每個鄰居節點發送更新後的距離向量。

所有節點不斷異步交換他們的距離向量，則估計開銷 $D_{x}(y)$ 會收斂到 $d_{x}(y)$，也就是實際的最低開銷路徑的開銷。

```
/* Initialization */
for all destinations y in N:
	D_x(y) = c(x, y) // if y is not a neighbor then c(x, y) = INT_MAX
for each neighbor w:
	D_w(y) = ? /* Have not recieved any neighbor cost table yet */
for each neighbor w:
	send distance vector D_x = [D_x(y) : y in N] to w

/* Loop */
loop
	wait (until a link cost change to some neighbor w 
	or recieve a distance vector from some neighbor w)
	
	for each y in N:
		D_x(y) = min_v{c(x, v) + D_v(y)}
		
if D_x(y) changed for any destination y:
	send distance vector D_x = [D_x(y) : y in N] to all neighbors
forever
```

在 DV 中，你只需要知道當前節點通向最短路徑的下一跳，也就是 $v^{*}(y)$ 。而 $v^{*}(y)$ 其實由 `D_x(y) = min_v{c(x, v) + D_v(y)}` 取到的 $v$ 決定，所以在處理這邊的同時，應該更新轉發表為 $v$。

### Distance Vector Algorithm: Link-Cost Changes and Link Failure

當 DV 算法檢測到節點與鄰居之間的鏈路開銷產生變化，就會更新自己的距離向量。如果最低開銷路徑的開銷發生變化，則向鄰居通知新的距離向量。

若鏈路錯誤的發送自己與目標的開銷，則會造成很大的問題。

![[Pasted image 20251202144425.png]]

考慮以下場景：同樣是 $x,y,z$ 三個路由器，本來 $c(x,y)=4$，但後來改成 $c(x,y)=60$。而 $c(y,z)=1$，$c(x,z)=50$ 沒有改變。因為 $y$ 檢測到開銷改變，所以會更新距離向量。因為 $z$ 還不知道 $c(x,y)$ 開銷的改變，所以 $D_{z}(x)=5$ 保持不變，$D_{y}(x)=c(y,z)+D_{z}(x)=1+5=6$，然後傳送自己的距離向量給鄰居，並設定下一跳為 $z$。

$z$ 收到 $D_{y}(x)=6$ 後，會被更新 $D_{z}(x)=6+1=7$，並設定下一跳為 $y$。導致封包在 $y,z$ 之間被傳來傳去，而且 $D_{y}(x)$ 與 $D_{z}(x)$ 的值不斷增加，直到開銷往復迭代後被正確更新。

這種問題稱為**路由選擇循環(routing loop)** (封包被傳來傳去)，會導致 **無窮計數(count-to-infinity)** (距離不斷自增) 問題。

### Distance Vector Algorithm: Adding Poisoned Reverse

剛才提到的那個簡單小場景可以通過使用**毒性逆轉(poisoned reverse)** 技術避免。

毒性逆轉的思想如下：若 $z$ 決定通過 $y$ 到達 $x$，就會告訴 $y$ $D_{z}(x)=\infty$。

當開銷更新時，雖然 $y$ 到 $x$ 的開銷為 $60$，因為 $D_{z}(x)=\infty$ 所以還是照走這條路送，然後將更新的距離向量傳給 $z$。此時 $z$ 會更新自己的距離向量 $D_{z}(x)=50$，然後傳給 $y$。$y$ 會更新距離向量 $D_{y}(x)=51$，然後傳給 $z$ 並決定轉發封包給 $z$，所以他會告訴 $z$ 自己的 $D_{y}(x)=\infty$。

需要注意的是，毒性逆轉並沒有解決實際的無窮計數問題，因為涉及到不只兩個直接相連的鄰居節點迴路無法直接通過毒性逆轉技術檢測到。

### A Comparison of LS and DV Routing Algorithms

在此，$N$ 為節點集合，$E$ 為邊的集合。

- 報文複雜性：因為 LS 要求每個節點都知道網絡中所有鏈路的開銷，所以需要發送 $O(|N| \ |E|)$ 個報文。而且一但一條鏈路的開銷改變，就必須向所有節點發送新的鏈路開銷。而 DV 要求每次迭代時都向自己的鄰居交換報文。當鏈路開銷改變時，DV 只在此開銷造成最低開銷路徑改變時才會傳播改變的鏈路開銷。
- 收斂(convergence)速度：對 LS 來說，$O(|N| \ |E|)=O(n^{2})$。而 DV 收斂較慢，而且可能會發生選擇環路貨無窮計數問題。
- 健壯性：對 LS，因為一個節點只計算自己的轉發表，計算在某種程度上是分離的，保證了一定的健壯性。但是對 DV 來說，因為一個節點的計算會被廣播給鄰居，然後在下次迭代時廣播給鄰居的鄰居，最終擴散到整個網絡。

- LS：
	初始化：將所有相鄰節點的路徑收錄。
	循環直到所有路徑最小值確定：從還未確定最短路徑的集合中，取出一個與源節點路徑最短者，並將其放入已確定最短路徑的集合中。透過鬆弛 (注意區分鬆弛操作與 Bellman-Ford 方程) 操作確定其他節點若通過該選定節點是否可以更快抵達目標節點。
- DV：
	初始化：將距離向量初始化為到達鄰居節點的距離。
	循環：一但接收到鄰居的距離向量，跑 Bellman-Ford 方程。如果通過某鄰居到達某節點的距離較短，則更新。

# Intra-AS Routing in the Internet: OSPF

就上述討論來說，我們將所有路由器都同質化了，所有路由器都運行同樣的算法，但實際情況可不能這樣。

因為以下兩點原因：
- 規模：當路由器數量龐大時，讓路由器彼此交換、存儲對方的數據成本極高。若使用 DV 算法，在這麼龐大的基數下可能無法收斂。
- 管理自治：不同 ISP 可能運行著不同的路由算法。

上述問題可以通過將路由器組織成**自治系統(Autonomous System, AS)** 來解決，每個 AS 包含由同一個單位控制的路由器群。

通常來說，一個 ISP 下的所有路由器加上鏈路就組成一個 AS。當然，也有些 ISP 將網絡切成許多 AS。每個 AS 都會有一個唯一的 ASN 來標示。

在一個 AS 內的路由器都運行著相同的路由選擇算法，且互相擁有彼此的信息。AS 中的路由算法稱為**自治系統內部路由選擇協議(intra-autonomous system routing protocol)**。

## Open Shortest Path First (OSPF)

**開放最短路優先(OSPF)** 是一種 AS 內部路由選擇協議。OSPF 是一種 Link-State 協議，使用洪流與 Dijkstra 來運行。

在 OSPF 中，每個路由器都會構建出整個 AS 的圖，然後運行 Dijkstra 算法找到到達所有子網的最短路。注意，其中鏈路開銷由管理員手動設置。

在 OSPF 中，路由器會把自己的訊息廣播到整個 AS，而不是只送給鄰居。而且，一但鏈路發生變化 (開銷改變或連接/中斷狀態改變)，路由器就會廣播鏈路狀態信息。就算鏈路狀態沒有發生變化，也會週期性 (最少每 30 分鐘) 的廣播鏈路狀態。

OSPF 通告包含在 OSPF 報文中，此種報文直接與 IP 承載 (上層協議值為 89)，所以 OSPF 協議必須自己實現 RDT 或鏈路狀態廣播等功能。

OSPF 通過 HELLO 報文檢查鏈路是否存活。

而 OSPF 協議允許路由器從相鄰路由器獲取其網絡範圍鏈路狀態的數據庫。

OSPF 有以下優點：
- 安全：可以鑑別 OSPF 路由器間的交換 (如鏈路狀態更新)。使用鑑別，僅有受信任的路由器可以參與 AS 內的 OSPF 協議。可以防止惡意注意錯誤信息進入路由表。
- 多條相同開銷路徑：當到達目的地的多條路徑具有相同開銷時，OSPF 允許使用多條路徑。可以用來幫助負載均衡。
- 對單播與多播路由選擇的綜合支持：使用 MOSPF 可以拓展 OSPF 實現多播拓展。
- 支持在單個 AS 中的層次結構：一個 OSPF 自治系統可以層次化的配置多個區域。每個區域各自運行自己的 OSPF 鏈路狀態路由選擇算法，區域內的所有路由器互相廣播鏈路狀態。而區域內通常有一或多台**區域邊界路由器(Area Border Router, ABR)** 負責向外轉發封包。一個 AS 會有一個區域被配製成**主幹區域(backbone area)**，負責連接所有其他區域。而所有 ABR 都屬於主幹區域，且所有 area 間的流量都必須通過主幹區域。

# Routing Among the ISPs: BGP

剛才我們討論過 AS 內部選擇協議，現在我們說說 AS 間的路由選擇協議。

喔對，你不用想也知道，不同的 AS 間應該都運行相同的 AS 間路由選擇協議。我們將其稱為**邊界網關協議(Border Gateway Protocol, BGP)**。

## The Role of BGP

在 BGP 中，分組不是路由到一個特定地址，而是路由到一個 CIDR 化的前綴，而每個前綴表示一個子網或子網集合。

BGP 提供以下功能：
- 從鄰居 AS 獲得前綴的可達性信息：BGP 允許子網向因特網通告自己存在，否則沒人可以到達該域。也就是說，BGP 幫忙告訴別人某個前綴存在於哪個 AS 中。
- 確定到某前綴的 "最佳" 路由：利用可達性信息與不同策略確定 "最佳" 路由，而不一定是最短的。

## Advertising BGP Route Information

上圖具有三個 AS，每個 AS 都有**內部路由器(internal router)** 與**網關路由器(gateway router)**。其中網關路由器位於 AS 邊緣，連接到其他 AS。內部路由器則只與所屬 AS 中的路由器通信。

考慮以下任務，通告前綴 $x$ 的可達性信息給所有其他路由器。讓我們用 high level 走一次，首先 AS3 對 AS2 通告 $x$ 在 AS3 中，我們將報文表示為 "AS3 x"。然後 AS2 會向 AS1 通告可以通過自己與 AS3 到達 $x$，報文表示為 "AS2 AS3 x"。

好啦，實際上 AS 不會自己互相傳東西，而是通過路由器。

在 BGP 中，路由器使用 `179` 端口與半永久 TCP 連接交換路由選擇信息。其中，跨越 AS 的連接稱為 eBGP 連接 (外部 BGP)，而 AS 中的連接稱為 iBGP 連接 (內部 BGP)。

需要注意的是，BGP 連接並不一定與物理鏈路對應，這是一種虛擬的連接。(就像 TCP 嘛)

所以，真實的通告過程如下。3a 向 2c 發送 "AS3 x"，2c 會向 AS2 中的路由器轉發此消息，然後由 2a 向 1c 發送 "AS2 AS3 x"，最後由 1c 發送給所有 AS1 中的路由器。

但是，我們可能有不同方法都可以到達 $x$ 呀！怎麼辦呢，我們下節討論～

## Determine the Best Routes

ok 呀，開始前咱先介紹點 BGP 術語。

當路由器通過 BGP 連接通告前綴時，還會附帶些 **BGP 屬性(BGP attribute)**。我們統稱前綴與其屬性為**路由(route)**。

兩個較為重要的屬性是 `AS-PATH` 與 `NEXT-HOP`：
- `AS-PATH`：包含通告已通過的 AS 列表，每當一個前綴通過一個 AS 時，該 AS 就會將自己加入 `AS-PATH` 中。為了防止環路，一個路由器在路徑列表中看到自己的 ASN，該路由器應該拒絕通告。
- `NEXT-HOP`：包含了 `AS-PATH` 中最起始的路由器接口的 IP 地址。例如，回想剛才的場景，則 AS1 中的每台路由器都會有 `NEXT-HOP` 為 2a 的 IP 地址。注意到， `NEXT-HOP` 屬性不可能包含 AS1 某路由器的 IP 地址。

就上述討論，每條 BGP 路由包含：`NEXT-HOP, AS-PATH, destination prefix`。當然，實際上不只這點屬性。

### 熱土豆路由選擇 (hot potato routing)

![[Pasted image 20251207223957.png]]

考慮上圖場景與路由器 1b。使用熱土豆路由選擇，選擇到達 `NEXT-HOP` 的路徑會具有最少開銷。例如，1b 到 2a 的開銷可能為 2，而到 3d 的開銷可能為 3，所以 1b 選擇通過路由器 2a。然後，找到可以通到 2a 的接口 I，並將 `(x, I)` 加入轉發表。

但是，對 1d 來說，可能直接選擇 3d，因為到 3d 的開銷為 1。

![[Pasted image 20251208012339.png]]

注意到：在轉發表中增加 AS 向外前綴時，AS 間與 AS 內部路由選擇協議都會用到。

熱土豆路由的思想是：尋找最低開銷(但不一定是最短路徑，與政策有關)並迅速將分組送出 AS，而不考慮分組在 AS 外會被如何傳遞，也不考慮之後的開銷。

也就是說，熱土豆路由是一種自私的算法，只顧著分組在 AS 內的開銷，而不思考 AS 之外的開銷。

注意到熱土豆路由選擇中，同一個 AS 中的相異兩台路由器可能會有不同的 AS 路徑，就像之前討論 1b 與 1d 那樣。

### 路由器選擇算法 

實際上，iBGP 是依序使用以下邏輯篩選路徑：
1. 路由被指定一個**本地偏好(local preference)** 作為其中一個屬性，由管理員設定。擁有最高偏好的路由會被選擇。
2. 若都有同一個偏好值，選擇具有最短 `AS-PATH` 者。此距離使用 DV 算法計算，而測度使用的是跳過 AS 的跳數，而非路由器跳數。
3. 使用熱土豆路由選擇，選擇能最快速到達 `NEXT-HOP` 的路徑。
4. 若還有路由，使用 BGP 標示符來選擇。

這樣看來，BGP 就不算是自私的算法了。因為在熱土豆前，咱先考慮了最短 `AS-PATH` 者。

## IP-Anycast

BGP 還常被用於實現 IP 任播 (anycast)，也就是說多個服務器共用一個 IP 地址，而封包會被路由到離用戶最近的服務器上。

我們假設 CDN 使用 IP 任播，則在配置階段 CDN 公司會給多台服務器指派相同的 IP 地址，並使用標準 BGP 從這些服務器通告。

當一台 BGP 路由器收到來自同一個 IP 地址的多個路由通告，它會將這些通告視為可以到達該 IP 地址的不同路徑。當路由器配置路由選擇表時，會根據本地運行的 BGP 算法選擇一個最優路徑。

但是實際上 CDN 通常不使用 IP 任播，因為 BGP 路由選擇變化可能使得同一 TCP 連接的分組到達不同的服務器。而 BGP 大多被廣泛運用於 DNS 系統。

## Routing Policy

我們已知路由策略是最高等級的規則，如果該 AS 不想轉發某來源的分組，就算該 AS 是該來源分組到達目的 AS 的最短路徑，BGP 也不會考慮該路徑進行轉發。

而這種行為可以通過控制 BGP 路由通告方式實現，比如不發送到達某目的地的通告，使得分組無法經由此 AS 到達目的地。就算實際上擁有一條物理鏈路連接到目的地，但由於屏蔽了通告，其他 AS 並不知道有這麼一條路徑可以到達目的地。

# The SDN Control Plane

回憶一波，SDN 體系結構具有 4 個關鍵特徵：
- 基於流轉發：與傳統轉發不同(IP 數據報的轉發只基於目的 IP 地址)，OpenFlow 1.0 可以允許 11 個不同首部字段值轉發。
- 數據平面與控制平面分離。
- 網絡控制功能：SDN 中，控制平面由兩個組件組成：SDN 控制器與網絡控制程序。控制器維護準確的網絡狀態信息並提供該信息與方法給控制程序，使得控制程序可以通過方法掌控下面的網絡設備。
- 可編程網絡。

## The SDN Control Plane: SDN Controller and SDN Network-control Applications

![[Pasted image 20251209175155.png]]

上面說過 SDN 分為兩層：控制器與網絡應用程序。

控制器的功能可分為三個層次：
- **通信層(communication layer)**：SDN 控制器和受控網絡設備之間的通信。控制器與受控設備之間的通信需要一個接口，我們稱之為**南向(southbound)** 接口。例如 OpenFlow。
- **網絡範圍狀態管理層(network-wide state-management layer)**：由 SDN 控制平面做出的最終控制決定 (如配置流表實現負載均衡)，會要求控制器具有相關網絡的主機、鏈路、交換機等信息。這些信息構成了 SDN 控制器維護的網絡範圍狀態。
- **給網絡控制應用程序層的接口(interface to the network-control application layer)**：控制器通過**北向(northbound)** 接口與網絡控制應用程序交互。該 API 允許程序在狀態管理層之間讀寫網絡狀態和劉表。

SDN 控制器被認為是邏輯上集中的，但實際上我們還是使用分布式服務器集合實現。

## OpenFlow Protocol

OpenFlow 協議運行在 SDN 控制器和受控的交換機或其他實現了 OpenFlow API 的設備之間。

OpenFlow 默認運行在 TCP 上，使用 `6653` 作為默認端口號。

從控制器流向受控交換機的幾個重要報文如下：
- `Configuration`：查詢或修改交換機的配置。
- `Modify-State`：修改交換機流表中的表項或端口特性。
- `Read-State`：從流表或端口收集統計數據或計數器值。
- `Send-Packet`：從交換機的特定端口發出特定的報文。

從受控交換機流向控制器的幾個重要報文如下：
- `Flow-Removed`：通知控制器自己刪除了一個流表項。可能由於超時或收到 Modify-State 報文。
- `Port-State`：通知端口狀態的變化。
- `Packet-in`：當一個分組無法與流表中任意一項表項匹配時，將這個分組發給控制器做額外處理。又或者某個匹配的流表項要求該分組被發給控制器。分組入報文被用於向控制器發送分組。

# ICMP: The Internet Control Message Protocol

**ICMP(因特網控制報文協議)** 被主機和路由器用來溝通彼此的網絡層信息。例如某個路由器無法找到一個通往指定 HTTP 請求的路徑，該路由器就會向你的主機發出一個 ICMP 報文以標明該錯誤。

ICMP 通常被認為是 IP 的一部分，但從體系結構上講它位於 IP  之上，因為 ICMP 是作為 IP 的有效載荷承載的。

ICMP 報文有一個類型字段和一個編碼字段，還包含了造成該 ICMP 報文產生的 IP 數據報的首部與前 8 個字節。

![[Pasted image 20251209212250.png]]

`ping` 程序會發送一個 ICMP 類型 8 編碼 0 的報文到指定主機，而主機會回應一個類型 0 編碼 0 的 ICMP 回顯回答。

`traceroute` 也是用 ICMP，他會向目的地主機發送一系列 IP 數據報，這些數據報都是一個不可達 UDP 端口號的 UDP 報文段。第一個數據報的 TTL 為 1，第二個為 2，以此類推。源主機在發送時也為數據報啟動定時器。當第 $n$ 個數據報到達第 $n$ 個路由器時，TTL 剛好過期。此時，路由器丟棄該數據報並發送一個 ICMP 報文 (類型 11 編碼 0) 給源主機，同時包含了路由器名稱與 IP 地址。此時源主機可以從定時器計算出 RTT。

當源主機收到端口不可達的報文段時，就會發送類型 3 編碼 3 的 ICMP 報文。此時 `traceroute` 就可以停止發送探測報文。

## The Simple Network Management Protocol

**簡單網絡管理協議(Simple Network Management Protocol, SNMP)** 是一個應用層協議，用於在管理服務器和代理間傳遞控制與信息報文。

SNMP 中最常使用的是請求響應模式，其中管理服務器向代理發送一個請求，代理接收並執行操作後，對該請求發送回應。

另一種很常被使用的功能是代理向管理服務器發送的一種非請求報文，稱為**陷阱報文(trap message)**。用於向管理服務器通知一個異常的發生。
